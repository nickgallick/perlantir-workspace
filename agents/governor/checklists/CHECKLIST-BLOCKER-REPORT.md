# CHECKLIST-BLOCKER-REPORT.md

**Purpose**: Verify blocker report is complete and actionable.

**When to Use**: Before reporting a blocker to operator.

---

## Report Completeness

- [ ] Exact operation blocked is named (not "system blocked")
- [ ] Layer where it failed is identified (1–4)
- [ ] Exact error message/symptom is included
- [ ] Root cause is explained (why this layer failed)
- [ ] Evidence of debugging is shown (commands + output)
- [ ] What work is blocked is stated (downstream impact)
- [ ] Options for resolution are listed (fix, alternative, defer)
- [ ] No soft language ("delayed", "on track", "will work when")

---

## Actionability

- [ ] Operator can understand blocker from this report alone
- [ ] Operator knows what needs to be done to resolve
- [ ] Operator can choose between specific options
- [ ] No information is missing that operator would need
- [ ] No ambiguity in blocker description

---

## Format

- [ ] Follows PLAYBOOK-BLOCKED-ENVIRONMENT template
- [ ] Is concise (not a novel, but complete)
- [ ] Uses exact technical language (no vague terms)
- [ ] Includes evidence (command + output)

---

**END CHECKLIST-BLOCKER-REPORT**
